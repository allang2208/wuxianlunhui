# 「无限轮回」游戏开发工作规则

> 生效日期：2026-07-17（取代 2026-06-21 旧版）
> 适用范围：本项目所有代码开发、美术资源整合、测试验证工作
> 目标：确保开发过程高效、安全、可回溯，避免引入回归性Bug
> 说明：本版已删除 legacy.js 单文件时代的全部过时内容（单文件架构禁令、legacy 备份命令、旧 DOM/CSS 限制、legacy 语境的对话迁移机制等）；架构原则以根目录 `WORKING-GUIDELINES.md` 为准。

---

## 一、最高优先级规则（2026-07-17 新增）

以下 4 条与任何历史文档中的旧规则冲突时，**以本节为准**，冲突的旧规则一律废止。

### 规则 1：能不硬编码就不硬编码
- 数值、尺寸、坐标、路径、颜色、开关等一律优先放入配置文件 / `data/*.json` / 常量模块，禁止散落在逻辑代码中。
- 同一数值只保留唯一真相源，引用处从配置读取。
- 修改既有公式/数值/常量仍需先输出拟改方案、经用户确认后方可执行（见规则 3.2）。

### 规则 2：开发功能前注意冲突和钩稽关系
- 动手前先排查新功能与哪些现有系统存在交集：数据、渲染、事件、生命周期、输入。
- 识别并维护钩稽（同步）链路，改动后必须同步链路另一端并验证两端一致。本项目已知链路示例：
  - `data/equipment.json` ↔ `public/data/equipment.json`（双份同步）
  - `data/enemy-config.json` ↔ 地牢工厂 `zombie-dungeon.js` ↔ 图鉴 ↔ `BootScene` 动画注册
  - 判定逻辑（torso-hitbox / skill-shapes）↔ 左下「范围」调试可视化（同一推导口径）
- 新系统接入前检查与既有系统的冲突点：事件名、实体 Map 键唯一性、深度排序、update 执行顺序。

### 规则 3：窗口贴图与实体生成固定显示统一使用 bottom: 固定像素
- 所有 UI 窗口/贴图以及实体生成后的固定显示定位，一律用 `bottom: 固定像素` 锚定。
- 禁止使用 top、百分比居中等相对方式摆放固定显示元素。
- （NPC 立绘此前已按此实践，现上升为强制规则。）

### 规则 4：素材先复制进项目子文件夹，再开展工作
- 实体动画、背景贴图等素材按用途在 `assets/` 下建立合理子文件夹（如 `assets/enemies/<name>/`、`assets/terrain/`）。
- 用户提供的素材必须先复制到项目对应子文件夹中，再开始改代码。
- 禁止直接引用项目外路径的素材文件。

### 规则 5：修改强绑定全场景生效
- 所有修复/改动必须落在共享链路上：修改后主神空间、地牢以及未来的所有场景统一生效。
- 禁止"仅单场景生效"的场景条件分支（如只在主神空间修复、地牢另行绕过）。
- 若某功能在不同场景表现不一致，优先排查共享状态是否被破坏（如贴图 active 标志、单例系统状态），从根因修复而非按场景打补丁。

## 二、开发工作流（Development Workflow）

### 2.1 迭代节奏
- **小步快跑**：每次修改聚焦一个明确目标，不跨系统大规模重构
- **增量交付**：每个功能完成后立即测试，不堆积多个修改后统一测试
- **前置规划**：复杂改动前必须口头/书面确认方案，不直接动手写代码

### 2.2 开发阶段
```
需求确认 → 方案设计 → 代码实现 → 本地测试 → 回归验证 → 交付汇报
```

| 阶段 | 要求 |
|------|------|
| 需求确认 | 明确修改目标、影响范围、预期效果 |
| 方案设计 | 说明改动思路，涉及哪些文件/函数/数据流；按规则 2 列出冲突点与钩稽链路 |
| 代码实现 | 遵循编码规范，保留关键注释 |
| 本地测试 | 启动 Vite 服务器，实际运行验证功能 |
| 回归验证 | 确认未破坏现有功能（装备系统、战斗系统、UI） |
| 交付汇报 | 说明修改内容、测试结果、已知问题 |

## 三、代码修改规则（Code Safety Rules）

### 3.1 修改前检查
- [ ] 执行备份：`node scripts/backup.js`
- [ ] 阅读目标文件上下文，理解现有逻辑
- [ ] 确认修改范围，评估是否会影响其他系统（规则 2）

### 3.2 修改中规范
- **不删除现有代码**：需要替换时，先注释旧代码，验证新代码正确后再删除
- **保留防御性检查**：`isFinite`、`null guard`、`naturalWidth` 等安全检查不可删除
- **事件绑定方式**：使用 `element.onmouseenter = fn` 直接赋值，不使用 `addEventListener`（防止重复绑定）
- **深拷贝传递**：装备/物品数据传递必须使用 `JSON.parse(JSON.stringify(item))`
- **对象池同步**：新增特效类型必须在 `EffectManager._factories` 中注册
- **公式和数值保护**：在未获得用户明确指示的情况下，不得修改现有的公式、数值和常量定义。涉及攻击力、防御力、速度、概率、成本等数值改动时，必须先输出拟改方案，经用户确认后方可执行。

### 3.3 修改后验证
- [ ] `npm run lint` 通过
- [ ] `npx vite build` 通过
- [ ] 涉及碰撞/命中判定时 `node scripts/test-collider.mjs` 通过
- [ ] 游戏能正常启动（菜单 → 开始游戏）
- [ ] 玩家WASD移动正常
- [ ] 鼠标瞄准和攻击正常（近战/弓/手枪）
- [ ] 空格闪避、Shift冲刺正常
- [ ] F键武器切换正常
- [ ] C键打开装备/背包面板正常
- [ ] ESC/点击关闭面板正常
- [ ] 装备栏右键装备/卸下正常
- [ ] 拖放系统正常（装备栏↔背包、装备栏内部交换）
- [ ] 图鉴系统正常打开
- [ ] 返回主菜单正常

### 3.4 禁止事项
- ❌ 使用 `file://` 直接打开 HTML（必须通过 `http://localhost:5173` 访问）
- ❌ 修改 `index.html` 中已有的 DOM 结构（除非新增面板/页面）
- ❌ 删除 `ItemDatabase` 或 `EquipManager.TEST_EQUIPMENTS` 中的现有条目
- ❌ 在逻辑代码中硬编码数值/路径/坐标（见规则 1）
- ❌ 引用项目外路径的素材（见规则 4）

## 四、文件与版本管理（File Management）

### 4.1 工作区结构
```
game-dev/
├── index.html          ← 游戏入口（修改需谨慎）
├── src/                ← 模块化源码（主要修改目标）
│   ├── config/ core/ systems/ entities/ combat/
│   ├── effects/ items/ ui/ world/ physics/
├── data/               ← 数据配置 JSON（唯一真相源）
├── public/data/        ← 与 data/ 同步的副本（Vite 开发服务器优先从此提供）
├── assets/             ← 游戏资源（按用途建子文件夹，见规则 4）
├── scripts/            ← 可复用脚本（一次性脚本归档到 scripts/archive/）
├── docs/               ← 文档目录
│   └── work-rules.md   ← 本文件
└── backup/             ← 自动备份目录
```

### 4.2 备份规则
- **每次修改前**：执行 `node scripts/backup.js` 创建带时间戳的备份
- **保留策略**：保留最近10个备份，超出时自动删除最早的

### 4.3 文件修改记录
每次修改后，在根目录 `CHANGELOG.md` 中按现有格式记录：
- 对话日期
- 修改的文件
- 修改内容摘要
- 测试结果
- 已知问题

## 五、测试与验证（Testing Protocol）

### 5.1 测试环境要求
- Node.js 已安装
- 启动命令：`npx vite`（在 `game-dev/` 目录下）
- 访问地址：`http://localhost:5173`
- **不可用浏览器直接打开文件**

### 5.2 核心功能测试清单（冒烟测试）
见 3.3 修改后验证清单，每次代码修改后必须执行。

### 5.3 浏览器控制台检查
- 测试完成后打开 DevTools（F12）检查：
  - 无红色报错（`console.error`）
  - 无重复警告（相同警告超过3次需关注）
  - 无 `undefined` 访问错误

## 六、需求与沟通规则（Communication）

### 6.1 需求确认流程
1. 用户提出需求 → 我复述理解 → 用户确认或修正
2. 复杂需求：先输出方案设计（不改代码），用户确认后再执行
3. 方案设计必须包含：修改文件、修改函数、预期效果、风险评估、钩稽链路（规则 2）

### 6.2 进度汇报格式
```
【任务】XXX
【状态】进行中/已完成/遇到问题
【修改文件】xxx.js（第X行 - 第Y行）
【测试情况】通过/未通过（附具体问题）
【下一步】XXX
```

### 6.3 问题上报规则
- 发现Bug时：说明复现步骤、期望行为、实际行为、相关代码位置
- 遇到技术限制时：明确说明原因，提供替代方案供用户选择
- 不隐瞒问题：即使问题不影响当前功能，也应记录在 CHANGELOG 中

## 七、质量门禁（Quality Gates）

### 7.1 提交前必须通过
| 检查项 | 标准 |
|--------|------|
| 静态检查 | `npm run lint` 通过 |
| 构建 | `npx vite build` 通过 |
| 碰撞单测 | 涉及判定时 `node scripts/test-collider.mjs` 通过 |
| 运行时稳定性 | 游戏能正常启动并运行30秒无崩溃 |
| 核心功能完好 | 冒烟测试清单全部通过 |
| 无回归Bug | 已修复的功能未被重新引入 |
| 浏览器兼容 | 在Chrome最新版中无报错 |

### 7.2 代码审查要点
- 是否有重复定义
- 是否有未声明变量（严格模式下会报错）
- 是否有 `div by 0` 风险
- 是否有 `null` 访问风险（`null.guard`）
- 事件监听器是否正确清理（内存泄漏风险）
- 是否新增硬编码（规则 1）
- 钩稽链路另一端是否已同步（规则 2）
