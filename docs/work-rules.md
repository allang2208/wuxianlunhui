# 「无限轮回」游戏开发工作规则

> 生效日期：2026-06-21  
> 适用范围：本项目所有代码开发、美术资源整合、测试验证工作  
> 目标：确保开发过程高效、安全、可回溯，避免引入回归性Bug

---

## 一、开发工作流（Development Workflow）

### 1.1 迭代节奏
- **小步快跑**：每次修改聚焦一个明确目标，不跨系统大规模重构
- **增量交付**：每个功能完成后立即测试，不堆积多个修改后统一测试
- **前置规划**：复杂改动前必须口头/书面确认方案，不直接动手写代码

### 1.2 开发阶段
```
需求确认 → 方案设计 → 代码实现 → 本地测试 → 回归验证 → 交付汇报
```

| 阶段 | 要求 |
|------|------|
| 需求确认 | 明确修改目标、影响范围、预期效果 |
| 方案设计 | 说明改动思路，涉及哪些文件/函数/数据流 |
| 代码实现 | 遵循编码规范，保留关键注释 |
| 本地测试 | 启动 Vite 服务器，实际运行验证功能 |
| 回归验证 | 确认未破坏现有功能（装备系统、战斗系统、UI） |
| 交付汇报 | 说明修改内容、测试结果、已知问题 |

## 二、代码修改规则（Code Safety Rules）

### 2.1 修改前检查
- [ ] 确认当前版本已备份至 `backup/` 目录（时间戳命名）
- [ ] 阅读目标文件上下文，理解现有逻辑
- [ ] 确认修改范围，评估是否会影响其他系统

### 2.2 修改中规范
- **不删除现有代码**：需要替换时，先注释旧代码，验证新代码正确后再删除
- **保留防御性检查**：`isFinite`、`null guard`、`naturalWidth` 等安全检查不可删除
- **事件绑定方式**：使用 `element.onmouseenter = fn` 直接赋值，不使用 `addEventListener`（防止重复绑定）
- **深拷贝传递**：装备/物品数据传递必须使用 `JSON.parse(JSON.stringify(item))`
- **对象池同步**：新增特效类型必须在 `EffectManager._factories` 中注册
- **公式和数值保护**：在未获得用户明确指示的情况下，不得修改现有的公式、数值和常量定义。涉及攻击力、防御力、速度、概率、成本等数值改动时，必须先输出拟改方案，经用户确认后方可执行。

### 2.3 修改后验证
- [ ] 游戏能正常启动（菜单 → 开始游戏）
- [ ] 玩家WASD移动正常
- [ ] 鼠标瞄准和攻击正常（近战/弓/手枪）
- [ ] 空格闪避、Shift冲刺正常
- [ ] F键武器切换正常
- [ ] C键打开装备/背包面板正常
- [ ] ESC/点击关闭面板正常
- [ ] 装备栏右键装备/卸下正常
- [ ] 拖放系统正常（装备栏↔背包、装备栏内部交换）
- [ ] 图鉴系统正常打开
- [ ] 返回主菜单正常

### 2.4 禁止事项
- ❌ 模块化拆分（必须保持单文件 `legacy.js` 架构）
- ❌ 使用 `file://` 直接打开 HTML（必须通过 `http://localhost:5173` 访问）
- ❌ 修改 `index.html` 中已有的 DOM 结构（除非新增面板/页面）
- ❌ 修改 `game-style.css` 中已有的选择器优先级（除非修复明确Bug）
- ❌ 删除 `ItemDatabase` 或 `EquipManager.TEST_EQUIPMENTS` 中的现有条目

## 三、文件与版本管理（File Management）

### 3.1 工作区结构
```
game-dev/
├── index.html          ← 游戏入口（修改需谨慎）
├── legacy.js           ← 游戏核心引擎（主要修改目标）
├── game-style.css      ← 样式表（UI调整时修改）
├── docs/               ← 文档目录
│   └── work-rules.md   ← 本文件
├── backup/             ← 自动备份目录
│   └── vYYYYMMDD_HHMMSS/  ← 时间戳命名的备份
└── assets/             ← 游戏资源目录
    ├── weapons/        ← 武器贴图
    ├── effects/        ← 特效贴图
    ├── icons/          ← 图标
    ├── items/          ← 物品掉落图
    └── ammo/           ← 弹药/投射物
```

### 3.2 自动备份规则
- **每次修改前**：创建带时间戳的完整备份（`index.html` + `legacy.js` + `game-style.css`）
- **备份命名**：`v20260621_143052/`（年-月-日_时-分-秒）
- **保留策略**：保留最近10个备份，超出时自动删除最早的

### 3.3 文件修改记录
每次修改后，在 `docs/CHANGELOG.md` 中记录：
```markdown
## 2026-06-21 14:30
- 修改文件：legacy.js
- 修改内容：修复G18切换后自动开火问题
- 影响范围：Player.switchWeaponMode()
- 测试状态：✅ 通过
- 修改者：AI Agent
```

## 四、测试与验证（Testing Protocol）

### 4.1 测试环境要求
- Node.js 已安装
- Vite 已安装（`npm install vite`）
- 启动命令：`npx vite`（在 `game-dev/` 目录下）
- 访问地址：`http://localhost:5173`
- **不可用浏览器直接打开文件**

### 4.2 核心功能测试清单（冒烟测试）
每次代码修改后必须执行：

```
□ 游戏启动流程
  □ 页面加载无控制台报错
  □ 菜单界面正常显示
  □ 点击"开始游戏"进入游戏

□ 移动与战斗
  □ WASD移动流畅
  □ 鼠标瞄准跟随
  □ 左键近战攻击（有动画、有伤害数字）
  □ F键切换到远程武器
  □ 左键远程攻击（弓/手枪）
  □ 空格闪避（有残影效果）
  □ Shift冲刺（有体力消耗）

□ 装备系统
  □ C键打开装备面板
  □ ESC关闭面板
  □ 右键背包物品装备到正确槽位
  □ 右键装备栏物品卸下
  □ 拖放装备栏内交换
  □ 拖放背包→装备栏装备
  □ 拖放装备栏→背包卸下

□ UI系统
  □ 顶部状态栏数据正确
  □ 底部状态条（HP/体力）正确
  □ 武器信息切换正确
  □ 图鉴页面正常打开
  □ 返回主菜单正常
```

### 4.3 浏览器控制台检查
- 测试完成后打开 DevTools（F12）检查：
  - 无红色报错（`console.error`）
  - 无重复警告（相同警告超过3次需关注）
  - 无 `undefined` 访问错误

## 五、需求与沟通规则（Communication）

### 5.1 需求确认流程
1. 用户提出需求 → 我复述理解 → 用户确认或修正
2. 复杂需求：先输出方案设计（不改代码），用户确认后再执行
3. 方案设计必须包含：修改文件、修改函数、预期效果、风险评估

### 5.2 进度汇报格式
```
【任务】XXX
【状态】进行中/已完成/遇到问题
【修改文件】legacy.js（第X行 - 第Y行）
【测试情况】通过/未通过（附具体问题）
【下一步】XXX
```

### 5.3 问题上报规则
- 发现Bug时：说明复现步骤、期望行为、实际行为、相关代码位置
- 遇到技术限制时：明确说明原因，提供替代方案供用户选择
- 不隐瞒问题：即使问题不影响当前功能，也应记录在 CHANGELOG 中

## 六、内存管理与对话迁移（Memory Management）

> ⚠️ 本项目 `legacy.js` 为 3000+ 行单文件，对话上下文累积后极易触发 token 长度限制。
> 当我检测到以下情况时，必须主动提醒用户执行「对话迁移」，避免上下文溢出导致无法响应。

### 6.1 触发提醒的条件（自查清单）

以下条件出现时，我必须在回复开头明确提醒用户：

| 条件 | 说明 |
|------|------|
| 🔴 多次读取 `legacy.js` 的上下文 | 已读取超过 3 个不同区段（`line_offset` 累计超过 2000 行） |
| 🔴 单次修改涉及超过 5 个独立函数 | 修改范围跨度过大，上下文累积严重 |
| 🔴 文件超过 100 行新增/修改 | 大段代码写入后，上下文膨胀 |
| 🟡 用户连续提出 5+ 个独立任务 | 每个任务都伴随文件读写，累积消耗 |
| 🟡 复杂调试（如反复读取文件找Bug） | 多次 `Grep`/`Read` 往返搜索，已读取大量文件内容 |
| 🟡 涉及生成大量新资源（如批量新增物品） | 新增数据量大，上下文占用增加 |

### 6.2 提醒格式

当触发条件时，我必须在回复中使用以下格式：

```
⚠️【内存自查提醒】
当前对话上下文已累积大量文件内容，建议执行「对话迁移」以释放内存。

【原因】简要说明触发条件（如：已读取 legacy.js 第1-500、1500-2000、2500-3000行，共约1500行上下文）
【建议】在当前对话中完成当前任务后，请说"新建对话继续"，我将迁移项目状态。
【当前进度】简述已完成的修改和未完成的任务。
```

### 6.3 迁移流程（用户说"新建对话继续"时）

**步骤 1：保存当前状态**
- 在 `docs/migration.md` 中写入迁移文档，包含：
  - 当前已完成的任务清单
  - 未完成的任务和进度
  - 最后一次测试通过的版本（备份目录名）
  - 已知的Bug或待办事项
  - 当前修改的文件的修改摘要

**步骤 2：告知用户迁移准备就绪**
```
✅ 迁移准备就绪

迁移文档已保存至：docs/migration.md
回退版本：backup/vYYYYMMDD_HHMMSS/

请新建对话后，发送以下指令，我将继续工作：
"继续开发无限轮回，读取 docs/migration.md 获取上下文"
```

**步骤 3：新对话中恢复上下文**
- 新对话中，用户发送指令后，我立即读取 `docs/migration.md`
- 读取项目转移汇报（如有）
- 读取最新版本的 `index.html`、`legacy.js`、`game-style.css`
- 确认当前状态后，按 `migration.md` 中的未完成任务继续工作

### 6.5 预防性措施

为减少迁移频率，我应主动执行：
- 优先使用 `Grep` 定位而非多次读取整段文件
- 修改文件前先用 `Bash` 复制备份，减少文件读取次数
- 复杂任务拆分为多个小任务，每完成一个小任务提醒用户是否继续
- 不重复读取已读过的文件区段（除非确认已修改）

## 七、质量门禁（Quality Gates）

### 7.1 提交前必须通过
| 检查项 | 标准 |
|--------|------|
| 语法正确性 | 无 JavaScript 语法错误 |
| 运行时稳定性 | 游戏能正常启动并运行30秒无崩溃 |
| 核心功能完好 | 冒烟测试清单全部通过 |
| 无回归Bug | 已修复的功能未被重新引入 |
| 浏览器兼容 | 在Chrome最新版中无报错 |

### 7.2 代码审查要点
- 是否有重复定义（如之前的 `EquipManager` 重复）
- 是否有未声明变量（严格模式下会报错）
- 是否有 `div by 0` 风险
- 是否有 `null` 访问风险（`null.guard`）
- 事件监听器是否正确清理（内存泄漏风险）
